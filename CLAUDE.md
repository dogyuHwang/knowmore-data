# knowmore-data 데이터 관리 가이드

이 문서는 Claude가 데이터 업데이트 요청을 받았을 때 따라야 할 기준과 절차를 정의합니다.
**데이터 업데이트 요청 시 반드시 이 문서 전체를 먼저 읽고 작업하세요.**

---

## MCP 도구 사용법

데이터는 JSON 파일을 직접 수정하지 않고 **MCP 도구를 통해** 읽고 씁니다.
MCP 서버는 `knowmore-mcp/server.py`에 구현되어 있으며, knowmore-data 레포를 직접 관리합니다.

| 도구 | 용도 |
|------|------|
| `list_topics` | 주제 목록과 현재 카드 수 확인 |
| `get_cards(topic_id)` | 특정 주제의 기존 카드 전체 조회 |
| `get_existing_words(topic_id)` | 기존 카드 단어 목록만 조회 (중복 방지용) |
| `replace_cards(topic_id, cards)` | 카드 전체 교체 (hot_issues 업데이트 시 사용) |
| `add_cards(topic_id, cards)` | 기존 카드에 새 카드 추가 (일반 주제 업데이트 시 사용) |
| `push_to_github(message)` | 변경사항을 GitHub에 커밋·푸시 |

### knowmore-mcp 연동

knowmore-mcp 서버(`../knowmore-mcp/`)는 데이터 관리 외에 **트렌드 조사** 기능도 제공합니다:

- `scheduler.py`: RSS 피드 + Naver DataLab API로 최신 트렌드를 수집
- hot_issues 업데이트 시 참고할 수 있는 검색량·뉴스 데이터를 제공
- MCP 서버 등록: `claude mcp add knowmore -- python ../knowmore-mcp/server.py`

이슈 카드 업데이트 시 MCP 도구의 `get_existing_words`로 중복 확인 + WebSearch로 트렌드 조사를 병행하세요.

## 카드 JSON 형식

```json
{
  "id": "ai_001",
  "topic": "AI",
  "word": "인공지능",
  "summary": "인간의 지능을 모방해 학습·판단·예측할 수 있도록 만든 컴퓨터 시스템",
  "description": "인공지능(AI)은 컴퓨터가 스스로 학습하고 문제를 해결할 수 있도록 만든 기술이에요. 얼굴 인식, 번역, 추천 알고리즘 등 일상 곳곳에 이미 쓰이고 있어요.",
  "example": "유튜브가 내 취향에 맞는 영상을 추천해주는 것도 인공지능 알고리즘 덕분이에요.",
  "level": "뉴비",
  "active": true
}
```

### 필드 설명

| 필드 | 필수 | 설명 |
|------|------|------|
| `id` | ✅ | 고유 식별자. `{topic_id}_{번호}` 형식 (예: `ai_001`) |
| `topic` | ✅ | 주제 이름. 위 표의 topic 값 사용 |
| `word` | ✅ | 카드 제목 (단어/용어). 10자 이내 권장 |
| `summary` | ✅ | 한 줄 요약. 40자 이내, ~했어요 말투 |
| `description` | ✅ | 본문 설명. 3~5문장, ~에요/~이에요 말투 |
| `example` | ✅ | 실생활 예시 문장. 자연스러운 대화체 |
| `level` | ✅ | 난이도: `뉴비` / `고인물` / `신` |
| `active` | ❌ | 생략 시 `true`. `false`면 앱에서 숨김 |
| `updatedAt` | ❌ | 이슈 분야 전용. 업데이트 날짜 `YYYY-MM-DD` |

## 난이도 기준

앱은 뉴비를 전부 학습해야 고인물이 해금되고, 고인물을 전부 학습해야 신이 해금되는 구조입니다.

| 값 | 해금 조건 | 기준 |
|----|----------|------|
| `뉴비` | 처음부터 열림 | 뉴스에서 자주 나오는 기본 개념. 처음 들어도 이해되는 수준 |
| `고인물` | 뉴비 전체 학습 후 | 배경 지식이 있어야 이해되는 내용. 구조나 원인까지 설명 |
| `신` | 고인물 전체 학습 후 | 전문가 수준의 심화 개념. 세부 메커니즘이나 파생 개념 |

### 난이도 판단 예시

- **뉴비**: "인플레이션", "GPT", "전세", "탄핵" — 뉴스 제목에 자주 등장, 설명 없이도 들어본 단어
- **고인물**: "테이퍼링", "RAG", "역전세", "헌법소원" — 뉴스 본문에서 다루는 개념, 배경 지식 필요
- **신**: "양적완화 출구전략", "파인튜닝 vs RLHF", "선순위 근저당", "위헌법률심판 제청" — 전문가들이 쓰는 세부 용어

## 이슈 분야 업데이트 (hot_issues)

이슈 분야는 **매주 1회** 업데이트합니다.

### 교체 규칙

- **최소 50% 이상** 새 카드로 교체
- **updatedAt이 3주(21일) 이상 지난 카드는 무조건 교체** (오래된 이슈 퇴출)
- 유지하는 카드의 updatedAt은 **변경하지 않음** (최초 생성일 유지 → 3주 카운트 기준)
- 새로 추가하는 카드만 updatedAt을 오늘 날짜로 설정

### updatedAt 규칙

- **최초 생성 시에만 설정**. 이후 업데이트 때 갱신하지 않음
- 유지 카드의 updatedAt을 갱신하면 3주 규칙이 무의미해지므로 절대 변경 금지
- 3주가 지나면 아무리 좋은 카드라도 교체해서 신선도 유지

### 업데이트 절차

```
1. 기존 카드 확인 → updatedAt 기준으로 3주(21일) 이상 된 카드 식별
2. 3주 초과 카드 전부 제거 대상으로 마킹
3. 나머지 중에서도 이슈성이 떨어진 카드 추가 제거 (최소 50% 교체 충족)
4. 2중 검증(MCP + WebSearch)으로 새 주제 조사
5. 새 카드 작성 (새 카드만 updatedAt = 오늘 날짜)
6. 유지 카드(updatedAt 그대로) + 새 카드 합쳐서 replace_cards로 교체
```

### 카드 수

- **최대 40개 이하**. 흥미로운 주제만 선별
- 수량 채우기 위해 뻔한 주제를 넣지 말 것
- 난이도 배분은 이슈 특성에 따라 자유롭게 판단

### 주제 선정 판단 기준

매 카드마다 아래 4가지 질문으로 판단한다:

1. **"이번 주에 새로 생긴 일인가?"** → Yes면 추가 후보
2. **"언제나 존재하는 상식인가?"** → Yes면 이슈가 아님, 제거
3. **"사용자가 궁금해할 만한가?"** → 밈, 트렌드, 생활 밀착 주제 우선
4. **"구체적인가?"** → "K-방산" 같은 포괄적 주제 대신 "천궁2" 같은 구체적 주제

### 좋은 주제 vs 나쁜 주제

| 좋은 주제 | 나쁜 주제 | 이유 |
|----------|----------|------|
| 석유 최고가격제 | 유가 폭등 + 오일쇼크 + 기름값 | 구체적 정책 1개 > 비슷한 주제 3개 |
| 천궁2 (96% 요격) | K-방산 | 구체적 무기 + 숫자 > 포괄적 카테고리 |
| AI 기본법 | AI 규제 | 특정 법률 > 추상적 개념 |
| 맥미니 대란 | 반도체 패권 | SNS에서 화제인 현상 > 뉴스 상투어 |
| 봄동 비빔밥 | 식료품 물가 | 바이럴 밈 > 늘 있는 주제 |
| 왕과 사는 남자 (1561만) | K-콘텐츠 | 구체적 작품+숫자 > 카테고리명 |

### 절대 포함하지 않는 주제

- **상시 존재**: 보이스피싱, 탄소 중립, 달러 패권, 가계 부채, 저출생 위기
- **포괄적 카테고리명**: K-방산, K-콘텐츠, 에너지 안보, 반도체 패권
- **이번 주 새 전개 없는 것**: 기후 위기, 고령화, 노인 빈곤
- **뉴스 상투어**: 글로벌 공급망, 수출 둔화, 공공요금 인상

### 반드시 포함할 주제 유형

- **이번 주 구체적 사건**: 법안 통과, 제품 출시, 기록 달성, 정책 시행
- **SNS/커뮤니티 화제**: 밈, 바이럴 트렌드, 인기 콘텐츠
- **생활 직격**: 새로 시행된 규제, 가격 변동, 신규 서비스 출시
- **문화/엔터**: 흥행 영화, 인기 드라마, 게임 출시, 아이돌 컴백
- **IT/기술**: 신제품 출시, 바이럴 서비스, 기술 사건 (주가 급등락 등)

### 주제 조사 방법 (2중 검증)

주제를 찾을 때 **MCP 트렌드 데이터 + WebSearch를 병행**한다. 한쪽만 의존하지 않는다.

```
[소스 1] knowmore-mcp (자동 수집 데이터)
  - scheduler.py가 RSS + Naver DataLab으로 수집한 트렌드 데이터
  - get_existing_words로 현재 카드 중복 확인
  - 검색량 기반이므로 "사람들이 실제로 검색하는 것"을 알 수 있음
  - 한계: RSS는 뉴스 위주, 밈/문화 트렌드는 잡지 못함

[소스 2] WebSearch (직접 조사)
  - MCP가 놓치는 밈, SNS 트렌드, 커뮤니티 화제를 직접 검색
  - MCP 데이터의 팩트 검증 (날짜, 수치, 구체적 사실 확인)
  - 개별 주제에 대한 심층 조사
```

**2중 검증 원칙**: MCP가 제안한 주제도 WebSearch로 팩트 확인. WebSearch에서 찾은 주제도 MCP의 검색량 데이터로 관심도 확인.

### 조사 소스 목록

| 소스 | 용도 | 접근 방법 |
|------|------|----------|
| **Naver DataLab** | 한국 검색량 트렌드 | knowmore-mcp scheduler.py (API) |
| **RSS 뉴스 피드** | 주요 뉴스 헤드라인 | knowmore-mcp scheduler.py (자동 수집) |
| **Google 검색** | 종합 뉴스 + 해외 시각 | WebSearch |
| **나무위키** | 사건 타임라인, 상세 정보 | WebSearch |
| **signal.bz** | 실시간 검색어 순위 | WebSearch |
| **loword.co.kr** | 네이버/구글 실시간 검색어 | WebSearch |
| **커뮤니티/SNS** | 밈, 바이럴, 문화 트렌드 | WebSearch (X, 클리앙, 에펨코리아 등) |
| **박스오피스** | 영화 흥행 순위 | WebSearch |
| **넷플릭스 TOP10** | 인기 드라마/영화 | WebSearch |
| **앱스토어/스팀** | 인기 앱/게임 | WebSearch |

### WebSearch 검색 전략

단일 검색으로 찾으면 뉴스 위주 결과만 나온다. **다각도 검색**으로 다양한 소스를 커버한다:

```
[1단계] 뉴스 전반 파악
- "2026년 {월} {주차} 한국 핫이슈 뉴스 종합"
- "2026년 {월} {일} 한국 뉴스 핫이슈 화제"
- "Korea trending topics {Month} 2026"

[2단계] 분야별 심층 검색
- 경제: "2026년 {월} 한국 물가 유가 환율 부동산"
- 정치/사회: "2026년 {월} 한국 법안 통과 정책 시행"
- IT/기술: "2026년 {월} 한국 IT 신제품 출시 앱 서비스"
- 문화: "2026년 {월} 넷플릭스 한국 인기 드라마 영화"
- 게임: "2026년 {월} 한국 인기 게임 출시"

[3단계] 밈/트렌드 발굴 (MCP가 놓치는 영역)
- "2026년 {월} 한국 화제 밈 트렌드 바이럴 SNS"
- 사용자가 언급한 키워드를 직접 검색: "{키워드} 2026"
- K-POP: "K-pop trending {Month} 2026 comeback"

[4단계] 개별 주제 팩트 검증
- 추가할 주제 각각에 대해 별도 검색으로 정확한 팩트 확인
- 날짜, 수치, 구체적 사실을 반드시 확인 후 카드 작성
- MCP 데이터와 WebSearch 결과가 일치하는지 교차 확인
```

### updatedAt

- 모든 카드의 `updatedAt`은 업데이트 당일 날짜(`YYYY-MM-DD`)로 설정

### ID 관리

- 매주 전면 교체이므로 `hot_issues_001`부터 순차 재번호
- 카드 수가 줄어도 됨. 001~036이면 036까지만 사용

### 카드 작성 실전 가이드

```
1. word: 제목만 보고 "뭔지 궁금하다"는 느낌이 들어야 함
   좋은 예: "램마겟돈", "봄동 비빔밥", "천궁2"
   나쁜 예: "글로벌 공급망", "에너지 안보", "수출 둔화"

2. summary: 40자 이내. 구체적 수치/사실 포함하면 좋음
   좋은 예: "DDR5 메모리 가격이 3개월 만에 4배 폭등했어요" (수치 포함)
   나쁜 예: "메모리 가격이 크게 올랐어요" (모호)

3. description: 3~5문장. "이게 왜 이슈인지"를 명확히 설명
   첫 문장: 무슨 일인지 (사실)
   중간: 왜 이런 일이 생겼는지 (원인/배경)
   마지막: 나한테 어떤 영향인지 (체감 영향)

4. example: 친구한테 직접 말하는 어투. 큰따옴표로 감싸지 않음. 실제 체험담처럼.
   좋은 예: X에서 강호동 봄동 비빔밥 영상 보고 바로 마트 갔는데 봄동이 다 팔렸더라고요.
   좋은 예: 맥미니 사려고 쿠팡 갔더니 품절이라 당근마켓까지 뒤졌는데 거기도 없더라고요.
   나쁜 예: "맥미니 품절 사태가 사회적 문제로 대두되고 있습니다." ← 뉴스 기사체, 금지
   나쁜 예: "맥미니를 사려고 했는데 어디서도 안 팔아서 당근마켓까지 뒤졌어요." ← 큰따옴표 감싸기, 금지

5. quiz.confusables: 외래어/고유명사에 추가
   "호르무즈 해협" → ["흐르무즈 해협", "호브루즈 해협"]
   "스태그플레이션" → ["스테그플레이션", "스태그인플레이션"]
```

### 카드 그룹핑 순서

ID를 순차 배정할 때 아래 순서로 그룹핑하면 가독성이 좋음:

```
1. 중동/에너지 관련
2. 경제/무역/금융
3. 안보/국제
4. 정치/사회/정책
5. 문화/엔터/트렌드
6. IT/AI/기술
```

### MCP 호출 순서

```
1. get_cards('hot_issues')          → 기존 카드 확인 (장기 유지할 이슈 파악)
2. WebSearch로 트렌드 조사           → 위 4단계 전략으로 다각도 검색
3. replace_cards('hot_issues', [...]) → 새 카드 전체 교체
4. push_to_github('이슈 카드 업데이트 YYYY-MM-DD')
```

## 일반 주제 업데이트 (hot_issues 제외)

### 카드 수

- 카드 수 상한 없음. 해당 분야에서 설명할 가치 있는 개념이라면 적극적으로 추가
- 중복 개념은 추가하지 않음 (`get_existing_words`로 먼저 확인)
- 난이도 배분은 주제 성격에 따라 자유롭게 판단

### 추가 기준

- 일반인이 뉴스·일상에서 마주칠 수 있는 개념
- 검색해도 쉽게 이해하기 어려운 것 (설명 가치가 있는 것)
- 너무 유사한 카드는 추가하지 않음

### 삭제/비활성화 기준

다음에 해당하는 카드는 `active: false` 처리하거나 삭제합니다.

- 현재 시점에서 더 이상 유효하지 않은 정보 (폐지된 제도, 사라진 서비스 등)
- 수치나 통계가 크게 달라진 경우 (업데이트 또는 삭제)
- 이미 해결된 일시적 이슈
- 더 좋은 카드로 대체 가능한 중복 개념

### ID 관리

- 삭제된 카드의 `id`는 재사용하지 않음 (앱 DB에 학습 기록이 남아 충돌 가능)
- 새 카드는 항상 기존 최대 번호 + 1부터 시작

### MCP 호출 순서

```
1. get_existing_words('topic_id')    → 중복 단어 확인
2. get_cards('topic_id')             → 필요 시 기존 카드 내용 확인
3. add_cards('topic_id', [...])      → 새 카드 추가
   또는 replace_cards(...)           → 대규모 정리 시 전체 교체
4. push_to_github('경제 카드 추가')
```

## 콘텐츠 작성 가이드

### 문체

- `description`: 구어체, ~에요/~이에요 말투. 3~5문장. 개념 설명 → 배경 → 한국과의 연관성 순서 권장
- `summary`: ~했어요 또는 명사형 종결. 40자 이내
- `example`: **친구한테 말하듯** 실생활 체험담. 큰따옴표 없이, "~더라고요", "~래요" 어투. 실제 있을 법한 구체적 상황

### 금지 사항

- 특정 정치인·기업·개인에 대한 주관적 평가
- 확인되지 않은 정보나 추측
- 영어 약어만 나열 (반드시 한국어 풀이 포함)
- 동일한 개념을 다른 단어로 중복 등재

### 앱 UI 고려

- `word`: 10자 이내 권장 (홈 카드 타이틀에 표시)
- `summary`: 40자 이내 (카드 미리보기에 표시)
- `description`: 5문장 이내 권장 (스크롤 길이 고려)

## 용어 사전 (glossary.json)

앱의 카드 설명/예시 텍스트에서 어려운 용어를 자동 감지하여 탭하면 쉬운 설명을 보여주는 **인라인 용어 사전** 기능에 사용됩니다.

### 파일 형식

```json
{
  "terms": {
    "양적완화": "중앙은행이 돈을 찍어서 시장에 푸는 정책",
    "테이퍼링": "양적완화를 서서히 줄여나가는 것",
    "ETF": "여러 주식을 한 바구니에 담아 거래하는 펀드"
  }
}
```

### 작성 가이드

| 항목 | 규칙 |
|------|------|
| **키 (용어)** | 카드 description/example 텍스트에 등장하는 어려운 용어 |
| **값 (설명)** | 한 줄 (40자 이내)로 쉽게 풀이. ~하는 것, ~하는 정책 등 명사형 종결 |
| **대상** | 경제/주식/부동산/법률/AI/IT/건강 분야의 전문 용어 |
| **매칭** | 한국어는 정확히 일치, 영어는 대소문자 무시 |

### 용어 추가 기준

- 카드 텍스트에 실제로 등장하는 용어 (등장하지 않으면 의미 없음)
- 일반인이 바로 이해하기 어려운 전문 용어
- 너무 쉬운 단어는 추가하지 않음 (예: "경제", "주식")

## 주의사항

- `id`는 전체 데이터에서 **유일**해야 합니다. 중복 시 앱 오동작 가능
- 삭제보다 `active: false` 처리를 권장합니다 (사용자 DB에 학습 기록이 남아 있을 수 있음)
- 앱은 **1시간마다** GitHub에서 새 데이터를 fetch합니다
- `뉴비` 카드를 추가하면 해당 주제의 해금 진행도가 초기화될 수 있으니 불필요한 추가는 자제합니다
- 카드를 삭제해도 사용자 DB의 학습 기록(`cardHistories`)은 남아 있으나, 앱 로직상 문제없이 동작합니다
